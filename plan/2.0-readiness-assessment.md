# whisper-transcribe 2.0 Readiness Assessment

**Date:** 2026-06-03
**Status:** Active strategy (not archived). Supersede or move to `plan/archive/` once 2.0 ships.
**Method:** Local verification commands + a multi-perspective read of every `js/` module, `index.html`, and `css/styles.css`, with adversarial verification of each surfaced defect (38 candidate defects reviewed → 29 confirmed real: 1 high / 11 medium / 17 low).

## Executive summary

whisper-transcribe is a healthy 1.0: green test suite (305 tests), clean lint and dependency checks, a 33 kB bundle, a fully-covered recording state machine, a clean model-adapter registry, and an event bus that genuinely decouples modules. The architecture is **not** what's holding 2.0 back. The real blockers are **user-trust defects in the record → transcribe → recover loop**: the recently-shipped retry feature is broken at the state-machine layer (and its test passes anyway), there is no request timeout (a hung request strands the app in `processing`), transcripts can be silently lost, the settings modal is not keyboard/AT-usable, and the repo lacks a LICENSE and honest setup/security docs. 2.0 is therefore **reliability-led**, with accessibility + UX polish as the immediate next track and release hygiene to close — no rewrite, no new dependencies.

## Definition of 2.0

> **2.0 = the record → transcribe → recover loop you can trust, that everyone can use, that is legally and operationally shippable.**
> Concretely: (a) no lost transcripts and no stuck states; (b) the retry/error paths behave correctly and provably; (c) WCAG-AA keyboard + screen-reader usable; (d) a real LICENSE, a version/release policy, and honest setup + security docs.

It is **not** a rewrite, a framework/bundler/TypeScript adoption, new transcription providers, or streaming. The raw-ES-module / no-build design is a feature and is retained.

## Findings by dimension

Tags: 🔴 2.0 blocker · 🟡 enhancer · ⚪ later.

### 1. Product / user experience — mixed
- **Strong:** single clean screen; pre-paint theme script avoids FOUC (`index.html:7-21`); live `aria-live` status; live visualizer.
- 🔴 Transcript lives only in the textarea `.value`, **never persisted** — reload/close/Cut wipes it with no undo (`ui.js:502-515`). Only action is a **destructive Cut** (`ui.js:145-173`) — no Copy, no Clear.
- 🟡 Re-recording appends with only `\n\n` (no timestamps/segments); Pause/Cancel are **always visible but inert when idle** (`index.html:160-189`); raw exception strings shown to users (`recording-state-machine.js:271-280`).
- **Fixes:** persist+restore transcript (S, 🔴); Copy + Clear alongside Cut (S, 🔴); segment boundaries (S, 🟡); hide Pause/Cancel until active (S, 🟡); friendly error copy + assertive errors (M, 🟡).

### 2. Recording lifecycle & state machine — mixed
- **Strong:** clean FSM, 100% covered; transition map in `constants.js:352-361`.
- 🔴 **HIGH — retry transition bug:** `retryPendingTranscription()` (`audio-handler.js:325-346`) runs `ERROR→IDLE` then `IDLE→PROCESSING`, but `IDLE→PROCESSING` is **not** a valid transition → it silently no-ops and logs *"Invalid state transition from idle to processing"*. During retry there is **no spinner, the mic stays enabled** (concurrent re-record possible), and a raw error is shown; the API call fires anyway, so it *sometimes* works. The retry test (`audio-handler-integration.vitest.js:308-318`) passes because it only asserts the end-state `IDLE`, never that `PROCESSING` was entered. Introduced in commit `f1504e3`.
- 🟡/⚪ Re-clicking mic while `STOPPING`/`CANCELLING` logs spurious invalid-transition errors (low); cancel can theoretically strand `CANCELLING` + leak the mic stream if `MediaRecorder`'s `stop` never fires (medium, speculative).
- **Fixes:** add `ERROR→PROCESSING`, do one transition in retry, honor `transitionTo` return values, and assert the `PROCESSING` phase in the test (all S, 🔴/🟡).

### 3. Model extensibility & provider architecture — mixed (enhancer)
- **Strong:** adapter pattern (`model-adapters/*`) with `buildRequest`/`parseResponse`/`storageKeys`.
- 🟡 Adding a provider still touches ~5–6 files (adapter + registry + `getModelConfig` + validation/`saveSettings` in `settings.js` + two `index.html` dropdowns + a settings DOM block) — the adapter hides ~30% of the work. `whisper-translate` is **registered + tested but unreachable from the UI**. Public `parseResponse()` depends on registry insertion order, not the active model (`api-client.js:245-256`).
- **Fixes:** drive credential storage from `storageKeys`; generate the model dropdown + settings sections from the registry; formalize/document the adapter contract; move key-format validation into the adapter (M/S, 🟡).

### 4. Settings / storage / configuration — mixed (enhancer)
- **Strong:** real validation (HTTPS, key charset, Whisper 32-hex) in `getValidationErrors` (`settings.js:543-575`).
- 🟡 API keys in **plaintext localStorage** (undocumented threat); `validateConfiguration()` is **dead code**; no schema versioning/migration; `settings.js` (734 lines) mixes API config with the sidebar drawer; `checkInitialSettings()` runs **twice** at startup (constructor `:79` + `ui.init` `:84`) due to construction order in `main.js`.
- **Fixes:** thin storage gateway (no deps, M); schema version + migration (S); reconcile the two validation paths (M); cover device/noise/sidebar branches (M).

### 5. Error handling & recovery — mixed
- **Strong:** retry/backoff for 429/5xx with `Retry-After` capping (`api-client.js:137-203`); offline pre-check.
- 🔴 **No `AbortController`/timeout** on the transcription fetch (`api-client.js:137-139`) — a hung request strands the FSM in `processing` with the mic disabled, **reload-only recovery**.
- 🟡 Retry button offered for **non-retryable errors** (401/403/400/bad config) — gates on blob presence, not retryability; `safeStopRecorder` failures emit `RECORDING_ERROR` that **no listener consumes**, stranding `stopping`; a single failure surfaces via **two independent channels** (triple-surfacing).
- ⚪ `permission-manager.js:228` references `MESSAGES.INVALID_REQUEST`, which **does not exist** in `constants.js` → a `TypeError` getUserMedia failure shows `undefined` (low, latent).
- **Fixes:** AbortController timeout (M, 🔴); gate retry on retryability (S, 🟡); consume `RECORDING_ERROR`→force `error` (S, 🟡); collapse to one message path (M, 🟡); add the missing constant (S).

### 6. Performance & runtime — enhancer
- **Strong:** tiny bundle; well-optimized visualizer (pre-quantized fill styles). Not a size problem.
- 🟡 **Synchronous WAV conversion** (`audio-converter.js`) on the main thread freezes the UI for hundreds of ms–seconds on long MAI-Transcribe recordings — the largest real runtime cost.
- ⚪ Visualization redraws full canvas at RAF rate despite 100 ms sampling; sampler runs when tab hidden; eventBus history allocates even when `debugMode` is off.
- **Fixes:** chunk/Worker the WAV encode (M, 🟡); gate redraw to new samples; pause sampler on `visibilitychange`; conditional history (S, ⚪).

### 7. Accessibility — mixed
- 🔴 Settings modal (`role="dialog"`) has **no focus trap, no initial focus, no focus restoration, and no Escape-to-close** (the Escape handler in `settings.js:281-292` only covers the sidebar) — keyboard users tab out behind the backdrop and can get stuck. Validation errors render **outside** the dialog and aren't announced. **Status text fails WCAG-AA contrast in both themes.**
- 🟡 **No `prefers-reduced-motion`** block (3 `@keyframes` always animate); `:focus-visible` defined for only a few inputs — core control buttons likely have **no visible focus ring**.
- **Fixes:** focus trap + Escape (M, 🔴); in-dialog announced errors (S, 🔴); contrast fix (S, 🔴); reduced-motion + `:focus-visible` for all controls (S, 🟡).

### 8. Test quality (not the %) — enhancer
- **Strong:** FSM + adapters genuinely well tested.
- 🔴 `settings-workflow.vitest.js` has **tautological tests asserting against their own mock** instead of production `checkRecordingPrerequisites`; `audio-handler-integration.vitest.js` has a MediaRecorder-`error` cleanup test for a **code path that doesn't exist**; the retry test passes despite the broken transition (§2). UI rendering + bootstrap are thin (`ui.js` 66% func).
- **Fixes:** replace tautological tests with real-method assertions (S, 🔴); fix/remove the dead-path test (M); add direct UI/bootstrap tests (M); de-alias integration spies (M).

### 9. Maintainability & module boundaries — enhancer
- **Strong:** event-bus decoupling; mostly cohesive modules.
- 🟡 `settings.js` conflates API config with the sidebar drawer; UI reacts to state via **two parallel mechanisms** — the `RECORDING_STATE_CHANGED` switch (`ui.js:206-245`) *and* granular `UI_*` events from state handlers (dual source of truth); some `APP_EVENTS` are dead; `ui.js` uses raw state-name string literals instead of `RECORDING_STATES`.
- **Fixes:** extract a `SidePanel` module (M, 🟡); collapse the duplicated prerequisite/config gate (S, 🟡); prune dead events + unify error channels (M, ⚪).

### 10. Documentation & release readiness — mixed
- 🔴 **No LICENSE file / no `license` field** → legally "all rights reserved", blocking forks/contribution. Version `1.0.0` with **no version/release policy** and no CHANGELOG. README doesn't cover **Azure resource creation or where to find the URI/key**, and **doesn't document the API-key-in-browser model** (no CSP, no SECURITY.md). README **misdescribes the git hooks**. Committed generated `docs/` (jsdoc) will go stale; CI is husky-only.
- **Fixes:** LICENSE + field (S, 🔴); version/release policy + tag 1.x (S, 🔴); expand README setup (M, 🔴); SECURITY.md + CSP (M, 🔴); CHANGELOG (S, 🟡); hosted CI mirroring husky (M, 🟡).

## Three-phase roadmap

**Phase 1 — Foundation / correctness & reliability.** Retry FSM fix (#1) · AbortController timeout (#3) · consume `RECORDING_ERROR` + gate retry on retryability + collapse error channels (#8) · fix `INVALID_REQUEST` · transcript persistence + Copy/Clear (#2, #5) · replace tautological tests + add bootstrap/UI tests (#9). *Make the core loop trustworthy and the suite honest.*

**Phase 2 — Product polish / accessibility / performance.** Modal focus trap/Escape/announced errors (#4) · contrast fix (#7) · `prefers-reduced-motion` + `:focus-visible` · hide inert Pause/Cancel + segment boundaries + friendly error copy · WAV off-main-thread + viz redraw gating. *Optional:* registry-driven model UI.

**Phase 3 — Release hardening / maintainability.** LICENSE + version policy + CHANGELOG (#6) · README setup expansion + SECURITY.md/CSP (#10) · hosted CI mirroring husky · settings schema versioning/migration · extract `SidePanel` from `settings.js` · prune dead events / unify dual-dispatch UI updates.

## Best first implementation goal

**Fix the retry state-machine transition.** Add `processing` as a valid target from `error` in `STATE_TRANSITIONS`, rewrite `retryPendingTranscription()` to do a single `ERROR→PROCESSING` transition, make transition callers honor the returned boolean, and **upgrade the retry test to assert the app actually enters `PROCESSING` (spinner shown, mic disabled) and logs no invalid-transition error.** Only HIGH-severity confirmed defect, small and contained, repairs the most recently shipped feature, and fixing the masking test raises the floor on everything else. Transcript persistence (#2) is the natural follow-up.

## Do-not-do-yet list

- ❌ No framework / bundler / TypeScript migration — keep raw ES modules.
- ❌ No new runtime dependencies (no XState, toast lib, a11y framework) — every item is doable with platform APIs.
- ❌ No streaming transcription yet — large effort vs Azure's batch model; revisit post-2.0.
- ❌ No backend/proxy for API keys yet — **document** the threat model first (Phase 3); a proxy is a separate architecture decision.
- ❌ Don't ship the `frontend-polish` visual wishlist (custom dropdowns, toasts, skeletons, OG tags) before correctness/reliability/a11y land.
- ❌ Don't chase 100% coverage — fix test *quality*, not the number.
- ❌ Don't over-split into micro-modules — only the clear `SidePanel` extraction is worth it.

## Decision log

- **2026-06-03 — 2.0 is reliability-led, not architecture-led.** The architecture (FSM, adapter registry, event bus) is already strong enough to carry 2.0; the highest-value risks are runtime correctness, recovery, transcript data loss, accessibility, and release readiness.
- **2026-06-03 — The repo will not be rewritten.** Incremental fixes within existing modules; the raw-ES-module / no-build design is retained.
- **2026-06-03 — No framework, bundler, TypeScript migration, backend proxy, streaming transcription, or new runtime dependencies** will be introduced for 2.0 unless separately justified later.
- **2026-06-03 — First implementation goal = the retry FSM fix** (`ERROR→PROCESSING`, honor transition returns, strengthen the retry test).
- **Sequencing:** strategy artifact (this doc) → retry fix → transcript persistence + Copy/Clear → request timeout + retryability gating → settings-modal accessibility pass.
